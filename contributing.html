Workflow

The basic workflow is as follows:

  1. Make sure the feature is something we want (ask in chat or whatever).
  2. Fork the master repository.
  3. Create a new feature branch called feature/XYZ (with XYZ being short name of the feature you're implementing), basing on develop.
  4. Work on your feature.
  5. Create a pull request.

Pull request MUST describe the feature you've implemented in detail. Your code will be reviewed, and you might be asked to make changes before the PR is accepted. Don't issue PRs on incomplete things.

Important: do not merge into master. We generally follow git-flow model, and the master is reserved for released code.

Commit messages

Follow tpope's guidelines for formatting the commit messages (no, the strict grammatical form is not required).

For an example, look at 2857b92c2a or 93c14ce17f.

Coding style

Generally PEP8. Look at the code and follow the lead, don't make me babysit you.

  • Follow SOLID principles.
  • LF line endings. If you edit on Windows, use a real fucking text editor and don't break my precious files.
  • ClassNames.
  • method_names, function_names, variable_names et cetera.
  • CONSTANT_NAMES.
  • 4 spaces indent. No tabs whatsoever.
  • Try to make lines not longer than 150 characters.
  • SPACES AROUND =! I will end you if you commit something like foo(x=1).
  • No dangling whitespace at the end of lines and in empty lines!
  • No line continuations! Use parentheses instead.
